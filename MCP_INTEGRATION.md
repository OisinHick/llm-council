# LLM Council MCP Integration Guide

This guide explains how to use the LLM Council with Model Context Protocol (MCP) tools for automated action execution.

## Overview

The 4-stage system allows you to ask the council to perform tasks, have them vote on the best approach, and then automatically execute that approach via MCP tools.

**Flow:**
1. **Stage 1**: All council members generate their approach/solution
2. **Stage 2**: Models vote/rank the approaches (anonymized peer review)
3. **Stage 3**: Chairman synthesizes all approaches and rankings
4. **Stage 4**: Based on voting results, generate and execute MCP tool calls

## Quick Start

### API Endpoint: `/api/action`

Send a JSON request with your task:

```bash
curl -X POST http://localhost:8001/api/action \
  -H "Content-Type: application/json" \
  -d '{
    "request": "Scan the network for open ports on 192.168.1.0/24 and save results to a file",
    "execute": true
  }'
```

### Streaming Endpoint: `/api/action/stream`

For real-time updates as the council deliberates:

```bash
curl -X POST http://localhost:8001/api/action/stream \
  -H "Content-Type: application/json" \
  -d '{
    "request": "Create a report file with system information",
    "execute": true
  }' \
  --stream
```

## Available MCP Tools

The council can use these tools to perform actions:

### 1. `execute_command` - Run Shell Commands

Execute any shell command (perfect for Kali Linux tools).

**Example request generated by council:**
```json
{
  "tool": "execute_command",
  "params": {
    "command": "nmap -sV 192.168.1.0/24",
    "timeout": 60
  },
  "description": "Scan network for services"
}
```

### 2. `read_file` - Read Files

Read file contents from the filesystem.

**Example:**
```json
{
  "tool": "read_file",
  "params": {
    "path": "/etc/hosts"
  },
  "description": "Read hosts configuration"
}
```

### 3. `write_file` - Write/Append Files

Create or modify files on the system.

**Example:**
```json
{
  "tool": "write_file",
  "params": {
    "path": "/tmp/scan_results.txt",
    "content": "Scan results...",
    "mode": "write"
  },
  "description": "Save scan results"
}
```

Or append to existing file:
```json
{
  "tool": "write_file",
  "params": {
    "path": "/tmp/scan_results.txt",
    "content": "\nAdditional results...",
    "mode": "append"
  },
  "description": "Append to results file"
}
```

### 4. `http_request` - Make API Calls

Send HTTP requests to external APIs.

**Example:**
```json
{
  "tool": "http_request",
  "params": {
    "method": "GET",
    "url": "https://api.example.com/data",
    "headers": {"Authorization": "Bearer token123"},
    "data": null,
    "timeout": 30
  },
  "description": "Fetch data from API"
}
```

## Example Use Cases

### Example 1: Network Security Scan

```bash
curl -X POST http://localhost:8001/api/action \
  -H "Content-Type: application/json" \
  -d '{
    "request": "Perform a comprehensive network scan on 10.0.0.0/24 and create a report with open ports and services",
    "execute": true
  }'
```

The council will:
1. Generate different scanning approaches (nmap, masscan, etc.)
2. Vote on the best approach
3. Synthesize the recommendation
4. Generate and execute the nmap command
5. Save results to a file

### Example 2: System Information Gathering

```bash
curl -X POST http://localhost:8001/api/action \
  -H "Content-Type: application/json" \
  -d '{
    "request": "Gather system information (OS, running services, open ports) and save to /tmp/sysinfo.txt",
    "execute": true
  }'
```

### Example 3: Vulnerability Assessment

```bash
curl -X POST http://localhost:8001/api/action \
  -H "Content-Type: application/json" \
  -d '{
    "request": "Check if common vulnerabilities are present: check SSH config, check sudo permissions, look for SUID binaries",
    "execute": true
  }'
```

### Example 4: Plan Without Execution

Get the council's recommended approach without executing it:

```bash
curl -X POST http://localhost:8001/api/action \
  -H "Content-Type: application/json" \
  -d '{
    "request": "What is the best way to perform privilege escalation testing on a Linux system?",
    "execute": false
  }'
```

This returns the action plan but doesn't execute it, letting you review and approve first.

## Python Client Example

```python
import httpx
import asyncio
import json

async def execute_council_action(task: str, execute: bool = True):
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://localhost:8001/api/action",
            json={"request": task, "execute": execute},
            timeout=300.0  # Long timeout for council deliberation
        )
        return response.json()

# Example usage
result = asyncio.run(execute_council_action(
    "Scan network for open ports and save to /tmp/results.txt",
    execute=True
))

# Check if all stages succeeded
if result.get("execution", {}).get("success"):
    print("✓ All actions executed successfully")
    for tool_result in result["execution"]["execution_results"]["results"]:
        print(f"  - {tool_result['tool']}: {tool_result['result']['success']}")
else:
    print("✗ Action execution failed")
    print(json.dumps(result, indent=2))
```

## How the Voting Works

1. **Stage 1**: Each model proposes a different approach
2. **Stage 2**: Models evaluate each approach anonymously (Response A, B, C, etc.)
3. **Aggregate Rankings**: Average position calculation determines the best approach
4. **Stage 4**: The best-ranked approach is transformed into executable MCP tool calls

This ensures:
- **No bias**: Models don't know which model proposed which approach
- **Collaborative**: Best ideas win based on merit, not the model's name
- **Transparent**: You can see all proposed approaches and the voting results
- **Actionable**: The top-voted approach is automatically executed

## Error Handling

If execution fails:

```json
{
  "stage4_action_plan": {
    "success": false,
    "error": "Could not find the best response"
  }
}
```

Or:

```json
{
  "execution": {
    "success": false,
    "error": "Action plan generation failed",
    "details": { /* ... */ }
  }
}
```

Review the error and try:
- Simpler, more specific request
- Check file permissions for file operations
- Verify command syntax for shell commands
- Check network connectivity for API calls

## Security Considerations

⚠️ **Important**: The council will generate and execute arbitrary commands. Use with caution:

- Only run with trusted models in the council
- Set appropriate file system permissions
- Review generated action plans before execution
- Use `execute: false` to review plans before running
- Run in isolated/sandboxed environments for sensitive operations
- Monitor execution results for unexpected behavior

## Customization

To modify available tools, edit `backend/mcp_tools.py`:
- Add new tool methods to `MCPToolExecutor` class
- Update the tool call handler in `execute_tools()` method
- Document the new tool parameters

To change how Stage 4 generates tool calls, edit `council.py`:
- Modify the `stage4_generate_action_plan()` prompt
- Adjust the JSON parsing logic
- Add validation for generated tool calls
